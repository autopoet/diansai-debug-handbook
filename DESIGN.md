# 电赛白皮书界面设计系统

这是整个应用的视觉与交互单一事实来源。页面可以有不同结构，但必须共用同一套字体、色彩、动效节奏和交互语法。

> 状态：创作、审核、富文本与评论闭环已落地 · 2026-07-23 复核
>
> 设计来源：`autopoet/my-blog` 与 `valaxy-theme-yun` 的结构方法，不复制其内容和组件

## 设计命题

界面是一册带个人署名、可以立刻检索的数字故障手册，不是后台管理系统，也不是由表单和待办堆成的工作台。

- 首页先出现“人”：未登录时是登录 / 注册入口，登录后是头像、名字、贡献与收藏。
- 搜索像频段：静止时只有一条干净的输入轨道，聚焦后才出现扫描、建议和明确动作。
- 内容像纸：下滑自然进入连续文档索引，阅读面稳定、不透明，适合长时间使用。
- 导航像卫星：位于页面边缘，需要时围绕当前页面展开，不占据整条顶栏。
- 动效有因果：只解释“身份出现、搜索被激活、内容可进入”。

首页借鉴 [`autopoet/my-blog`](https://github.com/autopoet/my-blog) 及其实际依赖的 `valaxy-theme-yun`：身份锚点、淡色雾面背景、衬线标题、细线分节、长留白、纵向入场线和向下内容流。参考源码中的主色是 `#0078E7`，主体文字是中性黑灰；不把社交图标里的紫色误当成全站主题色，也不复制头像、文案或组件代码。电赛白皮书在这套方法里保留真实搜索、协作身份和审核状态。

## 受众、任务与语气

- 受众：电子设计竞赛学生、知识贡献者、审核员和唯一管理员。
- 核心任务：从可观察的故障现象出发，快速找到下一步可执行检查。
- 语气：技术、编辑感、有探索性；不做营销式热闹，也不做静止的目录页。

## 结构家族

- 首页：`Identity Scroll / Index-First`。首屏围绕身份与核心搜索建立关系，下滑进入连续故障文档。
- 查 BUG：`Index-First`。搜索结果是高密度文档索引，不做筛选卡片墙。
- 文档：`Long Document`。正文是稳定纸面，目录、编辑、评论与版本信息围绕正文工作。
- 提交与审核：`Workbench`。Diff、版本责任人与审核动作处于同一工作面。
- 登录与个人页：沿用身份锚点、细线标签页和纸面文档流，不创造第二套系统。

## 视觉主题

所有实现值位于 `frontend/tokens.css`，组件只能引用 token。

- 底色：接近纸白的冷蓝灰；正文承载面可以使用纯白。
- 氛围：首页背景只允许极低对比的淡蓝、淡青和极弱粉雾与网格；它只能出现在背景，不能成为卡片填充。
- 墨色：冷黑与深灰，正文不使用低对比灰。
- 强调色：来源明确的信号蓝 `#0078E7`，视口占比不超过 5%；暖橙只可作为收藏状态的小标记。
- 状态色：绿色只代表审核通过，红色只代表删除或错误，不能承担装饰。
- 禁止玻璃模糊、彩色光球、大片渐变填充和纯装饰卡片。

## 字体

- 展示：`--font-display`，中文屏幕宋体序列，字重 700，正体；只用于品牌、身份名和主要文档标题。
- 正文与界面：`--font-body`，中文黑体序列，字重 400。
- 测量值、序号、版本：`--font-outlier`，等宽字体；只用于数据角色。
- 搜索、按钮、表单和正文不使用宋体；标题尺寸克制，不以超大字号制造“高级感”。
- 标题与正文必须形成明确的字体角色差；同一页面最多三套字体。
- 正文行长控制在 45–75 个中文字符的可读范围内。

## 导航

- 页面切换使用一颗不透明悬浮球：当前页面是核心球，点击、悬停或键盘聚焦后，其余入口沿弧线展开。
- 桌面端悬浮球位于左下；移动端位于底部中央并向上展开，避开安全区。
- 核心球使用纸面与中性墨色，信号蓝只出现在外圈、焦点和当前态；卫星节点保持纸面，不使用玻璃模糊或彩色光晕。
- 品牌固定在左上，仅包含图标和名称，不绘制整条顶栏。
- 全局搜索位于右上，只在当前页面没有主搜索时出现。
- 登录页只保留品牌。

## 搜索

- 首页搜索是页面的“故障频段”：默认由上下细线、搜索图标、输入文字和一个圆形动作组成，不绘制常见圆角白框。
- 聚焦时扫描线只经过一次，图标从静止变为定位状态，动作圆点向前推进；焦点环立即出现，不参与动画。
- 搜索框、登录注册输入框与登录模式标签共用“信号绽放”反馈：按下后控件轻微收紧，聚焦或选中后最多放大约 1%，低饱和浅蓝光晕从框内底部中心向上和两侧扩散并铺满整个控件背景，靠近边缘自然淡出；输入与搜索不绘制抢眼的完整矩形焦点框，依靠光晕、尺度与图标共同表达当前输入位置，Tab 仍保留紧凑的键盘焦点环。
- 登录模式标签不使用模板化蓝色下划线；选中态以文字权重、轻微尺度变化和底部中心光晕共同表达。
- 联想结果是唯一允许悬浮的纸面层，使用细边与单层阴影。
- 首页和查 BUG 页各有一个主搜索；全局搜索必须隐藏，避免重复。搜索占位文案只说一次，不增加提示标签和示例副标题。

## 动效

每页最多三种原语：

1. 身份出现：头像 / 标记、名字和动作在约 520ms 内完成一次编排入场。
2. 搜索激活：聚焦时扫描线单次经过，结果纸层随后出现；不无限循环。
3. 当前项反馈：文档行的箭头或细线移动 1–4px；不同时叠加缩放、旋转、阴影和变色。

只动画 `transform` 与 `opacity`。焦点环立即出现，不参与动画。`prefers-reduced-motion: reduce` 下取消空间位移，反馈不超过 150ms。

## 页面规则

### 首页

- 首屏不得出现营销副标题、功能卡片或待办式统计面板。
- 未登录时身份区只提供“登录”“注册”；登录后显示真实用户名、真实贡献数和真实收藏数。
- 主动作只有描述故障并搜索；搜索下方不重复同一句提示。
- 下滑后直接展示真实故障条目，条目总数与内容只能来自接口。
- 贡献与收藏必须跳转到个人页对应标签；文档收藏必须有真实数据闭环。

### 查 BUG

- 不使用卡片墙，不增加未实现筛选。
- 每条文档显示序号、标题、现象摘要和真实 ID。
- 文档行锚定在一条排障轨迹上；交互只突出当前行，不遮挡其他结果。

### 文档

- 桌面为目录、正文、阅读进度；移动端为单列。
- 正文使用稳定不透明纸面，行内编辑发生在原位置。
- 新文章可从编辑器动作一键插入标准排障模板；正文非空时只允许追加或取消，不静默覆盖。
- 提交前缺项提醒放在一次性的确认层中，不把编辑器长期改造成表单检查面板。
- 待审核版本在原文档位置提供“撤回并继续编辑”；未提交草稿可在二次确认后删除。
- 评论默认收起；选中同一正文块内的文字后才能出现评论动作。
- 评论线程持久化保存选区、上下文和版本锚点；支持回复、解决与重开。锚点失效时明确显示脱离状态，不伪造高亮。
- 版本必须显示修改者、审核者和日期。
- 正文末尾以两个行内动作提供“解决了 / 没解决”，不增加反馈卡片。
- 管理员治理动作收纳在文档的“更多”菜单中，只保留紧急撤下和从旧版本生成新回滚版本。

### 审核与版本

- 审核页先显示责任信息，再显示由 Tiptap JSON 转换出的可见语义块 Diff；审核员不应看到近似整行 JSON 的变化。
- Diff 提供“变更 / 修改前 / 修改后”三个紧凑视图，标题、段落、列表项、代码块、图片和表格行保持可读结构。
- 新增与删除必须同时使用符号、文字和颜色表达。
- 待审核版本不能覆盖公开正文。
- 管理员与审核员共享待审队列，允许审核自己的版本；第一位成功处理的人决定结果。
- 管理员的“发布官方种子”是独立次级动作，必须二次确认并明确显示发布来源，不能伪装成普通审核。

### 权限申请与通知

- 贡献者在个人页用一个小型对话框申请审核权，只填写熟悉方向与审核经验。
- 管理员在审核工作面的“审核权申请”标签处理，不新增用户管理后台。
- 已撤下文章只在同一工作面的“已撤下”标签出现，管理员可选择历史版本并填写原因生成回滚版本；不另建条目管理后台。
- 悬浮导航核心球旁使用一个安静的通知信号点；有未读时显示数量，点击后桌面展开纸面浮层、移动端展开底部对话框。
- 通知严格只有“新内容待审核、自己的投稿通过/驳回、评论收到回复”三类，不用额外文案和卡片填满面板。

## 实现映射

设计文档描述规则，代码文件承担具体实现。后续改界面时先修改规则，再修改对应实现，避免文档与页面再次分叉。

| --- | --- | --- |
| 全局颜色、字体、间距、动效 | `frontend/tokens.css` | 唯一允许声明原始颜色与字体的位置 |

| 主搜索与联想 | `frontend/src/components/search-form.*` | 首页与查 BUG 页复用；页面负责提供场景，不复制搜索框 |
| 首页身份与文档流 | `frontend/src/pages/home-page.*` | 身份、真实计数、主搜索和首屏文档索引 |
| 查 BUG 文档索引 | `frontend/src/pages/explore-page.*` | 关键词结果和连续文档行，不增加假筛选 |

| 个人身份、贡献、收藏与审核权申请 | `frontend/src/pages/profile-page.*` | 展示真实用户数据和最小权限申请，不变成账号管理后台 |
| 新建条目 | `frontend/src/pages/create-article-page.*` | 登录用户创建不可公开的首版草稿，并返回原文档继续编辑 |

## 组件契约

### 身份锚点

- 未登录：首页圆形标记、产品名、登录和注册；不显示虚构头像或统计。
- 已登录：首字母头像、用户名、真实贡献数和收藏数。
- 个人页沿用同一身份语法，不额外制造个人中心后台面板。

### 文档索引行

- 固定信息顺序：序号 → 标题 → 现象摘要 → 进入箭头。
- 标题是唯一强信息；摘要最多承担一层解释，不增加标签堆叠。
- Hover / Focus 只移动箭头或改变标题颜色，不整行浮起、缩放或发光。

### 状态反馈

- Loading：使用稳定骨架或短状态文字，不让布局跳动。
- Empty：说明为什么为空，并只给一个可执行入口。
- Error：说明失败对象并允许重试；不使用成功式彩色 Toast。
- Success：结果在页面中可见时保持静默，例如收藏状态直接切换。
- Disabled：同时使用原生 `disabled`、降低对比度和 `not-allowed` 光标。

## 当前覆盖与设计债务

### 已完成基线

- 首页、查 BUG、登录、个人页、文档页、新建条目和原位编辑已经使用当前 token 与结构规则。
- 收藏形成真实数据闭环；首页和个人页不再显示占位统计。
- 悬浮球、主搜索、全局搜索球已完成桌面和移动端验收。
- 新条目从草稿、提交审核到发布的状态闭环已接入真实接口；待审核内容不会提前公开。
- 文档页已经接入 Tiptap 结构化正文、图片上传、安全渲染和本地草稿快照。
- 划线评论已经接入持久化线程，支持回复、解决、重开、锚点恢复与脱离状态。

### 本轮设计目标

1. 写作闭环：标准模板、提交缺项提醒、撤回待审与删除未提交草稿都留在原文档编辑面。
2. 审核闭环：语义块 Diff、审核权申请、允许自审和官方种子发布共用现有 Workbench。
3. 可信度闭环：“解决了 / 没解决”、紧急撤下和可追溯回滚靠近正文与版本历史，不创建管理后台。
4. 通知闭环：用悬浮信号召回三类必要事件，不增加邮件、WebSocket 或通知分类设置。
5. 继续验收超长文档、复杂表格、图片和移动端；持续删除重复说明和无功能容器。

## 可访问性与响应式

- 必须检查 `320 / 375 / 414 / 768 / 1280 / 1440px`。
- `html` 与 `body` 使用 `overflow-x: clip`，不得用横向隐藏掩盖错误。
- 所有触控目标至少 `44×44px`，可点击文字不可换成两行。
- 每个悬停反馈都有 `:focus-visible` 或触控等价；焦点环对比度至少 3:1。
- 装饰线与节点使用 `aria-hidden`，图标按钮提供可读名称。

## 工程约束

- 使用语义 HTML、CSS Modules 和少量 React 状态，不引入只为动效服务的库。
- 不改动搜索、登录、编辑、审核等已有业务契约。
- 不写组件内临时颜色或字体；缺少值时先补 token。
- 功能不能回答用户任务时就删除，不用副标题、徽章或卡片填空。
- 前端改动至少通过 `npm run lint` 与 `npm run build`。
