# 电赛 Debug 协作知识库 PRD

> 文档状态：V0.1 草案  
> 暂定产品名：电赛白皮书  
> 目标版本：MVP  
> 技术栈约束：React + FastAPI + Peewee ORM  
> 更新日期：2026-07-19

## 1. 产品概述

### 1.1 产品定位

“电赛白皮书”是一个面向全国大学生电子设计竞赛参赛者、指导教师及相关技术爱好者的协作式 Debug 知识库。

网站以文档为核心组织内容，用户可以像使用维基百科一样共同创建、编辑和维护条目，也可以像使用飞书文档一样选中正文中的一段文字发起划线评论、回复讨论并解决评论。

产品重点不是提供泛化教程，而是沉淀电赛实践中的具体故障现象、排查路径、测量方法、根因和解决方案，让参赛者能在有限时间内快速定位问题。

### 1.2 核心价值

- 将散落在聊天记录、个人笔记和口头经验中的 Debug 知识结构化沉淀。
- 通过搜索、分类和标准化条目快速匹配“现象—原因—验证—解决方案”。
- 通过多人编辑、版本历史和讨论机制持续纠错，降低错误经验传播。
- 通过划线评论围绕具体内容讨论，减少“这句话哪里有问题”式沟通成本。

### 1.3 产品原则

- 先解决问题，再补充原理。
- 每个结论尽量可复现、可测量、可验证。
- 所有正文修改可追溯、可比较、可回滚。
- 开放贡献，但不允许无责任、无记录的匿名篡改。
- 对争议内容保留讨论上下文，不用覆盖历史的方式“解决分歧”。

## 2. 背景与问题

电赛 Debug 具有以下特点：

- 时间压力大，用户通常带着明确故障现象来搜索答案。
- 问题跨越电源、模拟、数字、射频、嵌入式、控制、通信、机械结构等多个方向。
- 相同现象可能由不同原因导致，单一“标准答案”往往不可靠。
- 有效经验通常依赖仪器、波形、参数、环境和操作步骤。
- 现有资料分散在博客、论坛、群聊、视频和个人笔记中，检索与验证成本高。
- 普通文档缺少社区治理，论坛又难以将讨论结果沉淀成持续维护的正文。

因此需要一种兼具“知识库正文”和“协作讨论”的产品形态。

## 3. 目标与非目标

### 3.1 MVP 目标

1. 用户能够快速找到与故障现象相关的知识条目。
2. 登录用户能够创建和编辑条目，并留下完整版本记录。
3. 用户能够选中文字发起评论、回复评论和解决评论。
4. 社区能够发现并处理错误、重复或低质量内容。
5. 管理员能够完成基本的内容治理和用户治理。

### 3.2 项目学习目标

本项目同时作为项目负责人的全栈学习与求职实践项目。负责人已有少量前端经验、暂时缺少后端经验，开发过程应兼顾产品交付和能力成长：

- 系统掌握 React 前端工程化、组件设计、状态管理和接口联调。
- 从基础开始学习 FastAPI、HTTP API、身份认证、权限控制和异常处理。
- 掌握 Peewee ORM、PostgreSQL、数据建模、事务、索引和数据库迁移。
- 学习版本审核、划线评论锚点等具有实际复杂度的业务建模。
- 建立单元测试、接口测试、代码规范、Git 协作、部署和监控能力。
- 最终形成可以写入简历、公开演示并在面试中完整讲解的全栈项目。

项目开发采用渐进式方式：每个阶段先理解关键概念和设计原因，再完成实现与测试，避免只追求功能数量而缺少对原理的掌握。

### 3.3 建议衡量指标

上线后前三个月的建议目标：

- 累计发布 100 篇有效条目。
- 至少 30 名有过编辑行为的贡献者。
- 搜索后进入某一条目的比例达到 50%。
- 进入条目后复制操作步骤、收藏或停留超过 60 秒的会话比例达到 30%。
- 被撤销或判定为无效的编辑占比低于 10%。
- 新评论在 72 小时内获得回复的比例达到 50%。

以上是初始运营目标，需在确定种子用户规模后调整。

### 3.4 MVP 非目标

以下能力不纳入首个版本：

- Google Docs 式多人实时光标和逐字实时协同。
- 原生移动端 App。
- 完整的问答社区、即时聊天或私信系统。
- 自动控制硬件、远程读取示波器或串口数据。
- 完整电路仿真、在线 IDE 或 PCB 在线编辑。
- 基于大模型自动生成并直接发布技术结论。
- 复杂积分、勋章、排行榜和商业化体系。

## 4. 目标用户

### 4.1 参赛学生

主要诉求：根据现象快速找到排查顺序和可执行方案；保存有用条目；补充自己验证过的经验。

典型场景：

- “稳压输出一带负载就掉压。”
- “ADC 采样值周期性跳变。”
- “电机闭环在目标值附近持续振荡。”
- “串口单独测试正常，整机运行后出现乱码。”

### 4.2 有经验的队员或往届选手

主要诉求：沉淀经验、纠正错误内容、参与技术讨论，形成个人贡献记录。

### 4.3 指导教师或领域专家

主要诉求：审核关键结论、补充理论依据、标记安全风险和不可靠方案。

### 4.4 内容管理员

主要诉求：维护分类结构，处理恶意编辑、垃圾内容、侵权内容和用户举报。

## 5. 核心使用流程

### 5.1 搜索并解决问题

1. 用户输入现象、器件名、模块名或错误信息。
2. 系统返回标题、摘要、标签和命中片段。
3. 用户进入条目，优先看到现象、快速检查清单和排查步骤。
4. 用户按步骤测量并验证。
5. 用户收藏条目、反馈“有帮助”，或评论仍有疑问的具体段落。

### 5.2 创建知识条目

1. 登录用户点击“创建条目”。
2. 输入标题，系统提示可能重复的已有条目。
3. 用户选择分类并填写标准模板。
4. 用户预览并提交，同时填写编辑说明。
5. 系统生成“待审核”版本，暂不向普通读者公开。
6. 审核员查看内容及提交说明，选择通过或驳回。
7. 审核通过后条目发布；驳回时必须填写原因，提交者可以修改后再次提交。

### 5.3 编辑与回滚

1. 登录用户进入条目并点击“编辑”。
2. 系统基于当前版本进入编辑状态。
3. 提交时填写本次修改说明。
4. 若正文已被他人更新，系统阻止直接覆盖并提示处理版本冲突。
5. 提交成功后生成待审核版本，审核期间线上仍展示最近一次已通过的版本。
6. 审核员对比当前公开版本和待审核版本，决定通过或驳回。
7. 审核通过后，新版本替换当前公开版本；驳回时保留提交记录及原因。
8. 用户可查看公开版本历史；提交者可查看自己的待审核或被驳回版本；有权限的用户可回滚。

### 5.4 划线评论

1. 用户在阅读模式中选中一段文字。
2. 页面展示“评论”浮动按钮。
3. 用户输入评论并提交。
4. 原文显示高亮，右侧评论区定位到对应讨论。
5. 其他用户可以回复、提及用户或将评论标记为已解决。
6. 正文更新后，系统尝试重新定位评论；无法可靠定位时标记为“原文已变更”，仍保留原引用和历史版本入口。

## 6. 信息架构

### 6.1 一级导航

- 首页
- 浏览分类
- 搜索
- 最近更新
- 创建条目
- 我的收藏
- 我的贡献
- 通知
- 管理后台（按权限显示）

### 6.2 故障现象主导航

知识发现和内容组织以“故障现象 → 排查步骤”为主线。用户不需要先判断问题属于哪个专业方向，即可从观察到的现象进入排查。

建议的一级故障现象包括：

- 无法上电、无法启动或反复复位
- 电压、电流或功耗异常
- 输出缺失、幅值异常或波形失真
- 噪声、纹波、漂移或数据抖动
- 采样不准、传感器数据异常
- 通信失败、丢包、乱码或时序错误
- 电机不转、失控、振荡或响应异常
- 器件过热、烧毁或保护触发
- 程序卡死、运行结果异常或性能不足
- 单模块正常、整机联调后异常
- 偶发故障、接触不良或环境相关问题

每个现象页优先展示快速检查清单，并引导用户按排查步骤逐层缩小问题范围。

### 6.3 技术方向与标签

- 电源与供电
- 模拟电路
- 数字电路与逻辑
- 单片机与嵌入式
- 传感器与信号采集
- 电机、驱动与控制
- 通信与协议
- 射频与无线
- FPGA
- PCB、焊接与硬件工艺
- 算法与上位机
- 仪器使用与测量方法
- 系统联调与赛场经验
- 安全与器件保护

技术方向作为辅助筛选，而不是用户寻找答案的唯一入口。标签用于器件、平台、比赛题型和具体特征等横向关联。

### 6.4 条目标准模板

每个 Debug 条目建议包含：

1. 故障现象
2. 适用范围与前置条件
3. 快速检查清单
4. 分步排查流程
5. 每一步的预期结果与分支
6. 可能根因
7. 对应解决方案
8. 如何验证已经修复
9. 风险与注意事项
10. 相关仪器、器件和环境
11. 图片、波形、代码或数据
12. 参考资料
13. 相关条目

正文编辑器应允许灵活增删章节，模板用于引导而非强制限制。

## 7. 功能需求

### 7.1 账号与权限

#### MVP

- 游客可以浏览、搜索和查看版本历史。
- 注册用户可以提交新条目或修改建议、评论、回复、收藏和举报，但内容必须审核后才会公开。
- 审核员可以审核新增条目和修改版本，审核时必须查看差异，驳回时必须填写原因。
- 管理员可以删除或恢复内容、回滚版本、锁定条目、处理举报、管理审核员和限制用户。
- MVP 使用用户名、邮箱和密码注册，登录时允许使用用户名或邮箱加密码。
- MVP 不接入手机号、微信、GitHub 等第三方登录。
- 用户主页展示简介、贡献记录和最近编辑。
- 所有提交、审核、驳回、编辑、回滚及管理操作记录操作者与时间。

#### 权限模型

| 能力 | 游客 | 注册用户 | 审核员 | 管理员 |
| --- | --- | --- | --- | --- |
| 浏览与搜索 | 是 | 是 | 是 | 是 |
| 查看公开版本历史 | 是 | 是 | 是 | 是 |
| 提交新条目或修改 | 否 | 是 | 是 | 是 |
| 评论与回复 | 否 | 是 | 是 | 是 |
| 审核内容 | 否 | 否 | 是 | 是 |
| 回滚版本 | 否 | 否 | 否 | 是 |
| 锁定条目 | 否 | 否 | 否 | 是 |
| 内容与用户治理 | 否 | 否 | 否 | 是 |

审核员不能审核自己提交的内容，避免自己提交、自己批准。后续可根据贡献质量邀请可信贡献者成为审核员。

### 7.2 条目阅读

- 展示标题、正文、目录、分类、标签、最后更新时间和贡献者。
- 支持标题锚点与目录跳转。
- 展示“最后验证时间”和“适用条件”，降低过期方案被误用的风险。
- 支持代码块、表格、公式、图片、文件附件和提示块。
- 支持相关条目和反向链接。
- 支持收藏、分享、反馈是否有帮助。
- 对涉及高压、大电流、电池、激光等内容展示安全提示。
- 游客可以阅读正文与公开评论，但必须登录后才能划线评论或回复。

### 7.3 编辑器

建议 MVP 采用成熟的结构化富文本编辑器内核，如 Tiptap/ProseMirror，而非自行开发编辑器。

MVP 只向用户提供类似飞书或 Word 的富文本编辑体验，不提供 Markdown 源码编辑模式。正文底层仍保存为结构化 JSON，未来如有明确需求可增加 Markdown 导入和导出。

必须支持：

- 标题、段落、列表、引用、表格、分割线。
- 行内代码和代码块。
- 图片上传、说明文字和替代文本。
- LaTeX/数学公式。
- 链接到站内条目和外部资料。
- 自动保存本地草稿。
- 提交前预览。
- 提交时填写编辑说明。
- 防止两个编辑者无提示地互相覆盖。

MVP 附件规则：

- 允许上传 JPG、PNG、WebP 图片，用于电路、波形、仪器读数和故障现场说明。
- 单张图片默认不超过 10 MB，服务端统一校验并生成适合网页展示的版本。
- 代码使用正文中的代码块，不单独上传源码文件。
- 暂不允许上传 ZIP、可执行文件、完整工程和其他任意格式附件。

MVP 不要求多人同时实时编辑。同一条目采用乐观并发控制：提交时校验用户开始编辑所基于的版本号，版本不一致时要求用户刷新、复制自己的修改并重新合并。

### 7.4 版本历史

- 每次正文提交生成不可变更的 Revision。
- 版本状态包括待审核、已通过和已驳回。
- 版本记录包括作者、时间、编辑说明、正文快照、来源版本、审核人、审核时间和审核意见。
- 普通读者只能查看已通过版本；提交者可查看自己的待审核与已驳回版本；审核员和管理员可查看全部版本。
- 支持以行级或块级差异展示两个版本。
- 管理员回滚时仍生成一个新版本，不删除原历史。
- 删除条目采用软删除，允许管理员恢复。
- 条目标题、分类和标签的变化也应进入版本记录或审计日志。

### 7.5 内容审核

- 新条目的首个版本和已有条目的每次修改都必须进入审核队列。
- 待审核修改不能改变当前公开内容。
- 审核页必须展示提交说明，以及与当前公开版本的差异。
- 审核结果为“通过”或“驳回”；驳回必须填写具体原因。
- 审核员不能审核自己提交的版本。
- MVP 只有项目负责人一名审核员，一人通过即可发布。
- 若审核期间公开版本发生变化，原待审核版本标记为“基线已更新”，需重新合并后提交，不能直接覆盖新版。
- 同一条目默认只允许一个有效的待审核修改，MVP 暂不实现多个修改提案的自动合并。
- 审核通过、驳回和撤回均产生通知和审计记录。

### 7.6 划线评论

#### 交互要求

- 阅读模式下选中文字后出现评论入口。
- 被评论文本显示高亮；点击高亮打开对应评论线程。
- 右侧评论面板按正文位置排列。
- 评论支持回复、提及、编辑自己的评论、解决与重新打开。
- 用户可筛选“未解决”“已解决”“与当前段落相关”的评论。
- 通知评论创建者、被提及者及线程参与者。

#### 锚点策略

每条划线评论至少保存：

- 创建评论时的文档版本 ID。
- 选中文本原文 `exact`。
- 选区前后的上下文 `prefix`、`suffix`。
- 结构化文档中的起止位置。
- 所属内容块 ID（若编辑器模型支持稳定块 ID）。

正文更新后按以下顺序重新定位：

1. 稳定块 ID 与块内位置。
2. 原文及前后文精确匹配。
3. 原文近似匹配。
4. 无法可靠匹配时标记为“已脱离当前正文”。

脱离正文的评论不被删除，用户仍可查看原引用、创建时版本和讨论内容。

#### MVP 简化建议

首版可以将评论绑定到创建时 Revision，并在当前正文中做精确文本重定位。近似匹配和复杂位置映射放到后续版本，以降低编辑器与锚点系统的实现风险。

### 7.7 搜索与发现

#### MVP

- 按标题、正文、标签、分类搜索。
- 结果显示标题、摘要、命中片段、分类、标签和更新时间。
- 搜索结果优先匹配故障现象和排查步骤，再匹配技术方向、器件和平台。
- 支持按故障现象、技术方向、标签和更新时间筛选。
- 输入标题创建条目时进行重复内容提示。
- 首页首先展示常见故障现象入口，其次展示技术方向、热门条目和最近更新。

数据库可优先选择 PostgreSQL，并使用其全文检索能力；数据规模和检索需求增长后再评估 Meilisearch 或 Elasticsearch。

### 7.8 内容治理

- 用户可以举报条目、版本和评论。
- 举报原因包括技术错误、危险操作、垃圾内容、侵权、恶意编辑和其他。
- 管理员可查看待处理举报及上下文。
- 管理员可回滚恶意版本、锁定条目、隐藏评论或限制账号。
- 锁定条目后普通用户不可编辑，但仍可评论和提交修改建议。
- 所有管理操作写入审计日志。

### 7.9 通知

- 自己提交的内容审核通过或被驳回。
- 自己提交的版本因基线更新需要重新合并。
- 自己的编辑被回滚。
- 自己的评论收到回复。
- 在评论中被提及。
- 参与的评论线程被解决或重新打开。
- 收藏的条目发生重要更新。

MVP 先实现站内通知；邮件通知作为后续能力。

## 9. 核心数据模型

以下为产品层面的建议模型，字段以最终技术设计为准。

### 9.1 用户与权限

- `User`：账号、昵称、简介、状态、创建时间。
- `Role` / `UserRole`：角色及用户角色关系。
- `AuditLog`：关键操作、操作者、对象、时间和操作详情。

### 9.2 知识内容

- `Article`：条目稳定 ID、当前版本、状态、锁定状态、创建者。
- `ArticleRevision`：正文快照、标题快照、作者、编辑说明、父版本、审核状态、创建时间。
- `RevisionReview`：审核人、审核结果、审核意见和审核时间。
- `Category`：树形分类。
- `Tag`：标签。
- `ArticleTag`：条目与标签关系。
- `Attachment`：图片或文件、上传者、类型、大小和存储地址。
- `ArticleRelation`：相关条目及关系类型。

### 9.3 评论

- `CommentThread`：所属条目、创建版本、锚点、状态、创建者。
- `Comment`：所属线程、回复对象、正文、作者、编辑与删除状态。
- `Mention`：评论与被提及用户关系。

### 9.4 用户行为

- `Favorite`：用户收藏。
- `HelpfulVote`：是否有帮助。
- `Report`：举报对象、原因、说明和处理状态。
- `Notification`：接收者、类型、关联对象、已读状态。

## 10. 技术方案约束与建议

### 10.1 前端

- React + TypeScript。
- 推荐使用 Vite 构建。
- 编辑器推荐 Tiptap/ProseMirror。
- 服务端状态管理可选 TanStack Query。
- 路由可选 React Router。
- 划线评论需要将 DOM 选区转换为编辑器文档位置及文本锚点。

### 10.2 后端

- FastAPI 提供 REST API。
- Peewee 负责关系数据库访问和事务。
- Pydantic 用于请求与响应模型。
- 数据库优先 PostgreSQL；本地开发可使用 SQLite，但不能以 SQLite 的行为代替 PostgreSQL 上线验证。
- 身份认证建议使用短期 Access Token + 可撤销的 Refresh Token，或安全的服务端 Session。
- 图片和附件使用对象存储；开发环境可使用本地兼容实现。

### 10.3 内容格式

推荐将编辑器的结构化 JSON 作为正文主存储格式，同时生成用于搜索的纯文本。不要只保存 HTML，因为后续进行评论锚定、结构化差异比较和内容迁移会更困难。

每个 Revision 保存完整正文快照。MVP 阶段优先保证正确性和可回滚性，暂不通过增量补丁节省存储。

### 10.4 并发与一致性

- 编辑提交必须携带 `base_revision_id`。
- 服务端在事务内校验当前版本。
- 版本冲突返回明确的 `409 Conflict`。
- 创建 Revision 与更新 Article 当前版本必须在同一事务完成。
- 评论创建时校验对应 Revision 与锚点数据。

### 10.5 Peewee 使用注意事项

- 请求生命周期内获取并释放数据库连接。
- 对创建版本、更新当前版本、写审计日志等组合操作使用事务。
- 为外键、唯一约束、状态筛选和时间排序建立索引。
- 数据库迁移需引入 Peewee 兼容的迁移方案，禁止依赖运行时自动建表管理生产结构。
- FastAPI 的异步接口与同步 Peewee 混用时，数据库操作需放入线程池，或统一采用同步路由处理数据库密集型请求，避免阻塞事件循环。

## 11. 非功能需求

### 11.1 性能

- 常规页面首屏在正常校园网络下 3 秒内可用。
- 搜索请求 P95 响应时间目标低于 800ms。
- 普通条目读取接口 P95 响应时间目标低于 500ms。
- 图片采用压缩、缩略图与懒加载。

### 11.2 安全

- 密码必须使用成熟的慢哈希算法保存。
- 所有写接口进行身份、权限和 CSRF/Token 校验。
- 富文本输出必须进行严格的 XSS 防护。
- 上传文件校验类型、大小和文件名，禁止可执行文件直接访问。
- 对登录、注册、搜索、评论和编辑接口进行速率限制。
- 重要管理操作要求二次确认并记录审计日志。
- 对危险电气操作设置醒目的安全提示与免责声明。

### 11.3 可靠性

- 数据库定期备份，并验证恢复流程。
- Revision 不允许物理覆盖。
- 附件删除采用延迟清理，避免历史版本引用失效。
- 关键写操作具备错误追踪与结构化日志。

### 11.4 可访问性与兼容性

- 支持桌面端 Chrome、Edge、Firefox 的近期稳定版本。
- 阅读页面适配移动端；编辑体验 MVP 以桌面端为主。
- 关键操作支持键盘访问。
- 图片支持替代文本，颜色不作为唯一状态提示。

## 12. API 模块建议

- `/auth`：注册、登录、刷新、退出。
- `/users`：用户信息与贡献。
- `/articles`：条目创建、读取、更新、状态管理。
- `/articles/{id}/revisions`：版本提交、列表、详情、比较与回滚。

- `/categories`、`/tags`：分类与标签。

## 13. MVP 验收标准

满足以下条件可认为 MVP 达到可内测状态：

1. 游客能浏览分类、搜索并完整阅读条目。
2. 用户能注册登录，提交新条目和已有条目的修改。
3. 未审核内容不会出现在公开阅读页面，原公开版本不受待审核修改影响。
4. 审核员能够查看版本差异并通过或驳回，且不能审核自己的提交。
5. 驳回必须包含原因，提交者能够修改后重新提交。
6. 每次提交和审核都会生成可追溯的版本或审核记录。
7. 两位用户基于同一旧版本提交时，后提交者不会静默覆盖先提交者。
8. 管理员能够比较版本并回滚，回滚行为本身形成新版本。
9. 用户可以选中正文文本创建评论，并在刷新页面后保持关联。
10. 评论支持回复、解决和重新打开。
11. 修改正文后，系统能重新定位完全一致的引用；无法定位时保留原引用并显示脱离状态。
12. 用户可以举报内容，管理员可以处理举报并查看操作记录。
13. 用户可以从故障现象进入条目，并按排查步骤逐层定位问题。
14. 核心流程在桌面端主流浏览器中可正常完成。
15. 富文本、链接和上传内容经过安全校验，不执行用户注入脚本。
16. 数据库备份与至少一次恢复演练成功。

## 15. 主要风险与应对

### 15.1 冷启动内容不足

应对：上线前邀请往届选手和指导教师，以高频问题为主题准备首批 30–50 篇种子条目。

### 15.2 开放编辑导致低质量或恶意内容

应对：所有新增和修改必须经过审核后才公开，同时要求登录、保留版本、限流、举报、回滚、锁定和审计。

### 15.3 错误方案造成器件损坏或人身风险

应对：安全分类与警示块、危险内容举报、高风险内容审核、适用条件和验证依据必填引导。

### 15.4 划线评论在编辑后漂移

应对：保存版本、原文、上下文和结构位置；无法匹配时诚实展示“原文已变更”，绝不错误绑定到相似段落。

### 15.5 编辑器开发成本失控

应对：基于成熟编辑器框架，MVP 不做实时协同，优先保证内容结构、版本和评论锚点稳定。

### 15.6 Peewee 与异步 FastAPI 的阻塞问题

应对：尽早统一数据库调用模式，进行并发压测，不在异步事件循环中直接执行同步数据库查询。

## 16. 已确认与待确认的产品决策

### 16.1 已确认

- 网站是完全公开的知识库，暂不支持学校或队伍私有空间。
- 注册用户可以提交新增或修改，但必须审核通过后才会公开。
- MVP 阶段只有项目负责人担任审核员，一名审核员通过即可发布。
- MVP 使用用户名、邮箱和密码，不接入手机号或第三方登录。
- 游客可以阅读，登录用户才能划线评论和回复。
- MVP 只允许上传 JPG、PNG、WebP 图片，单张默认不超过 10 MB；暂不开放工程文件和压缩包。
- 编辑器 MVP 采用富文本模式，不要求用户学习 Markdown。
- 知识结构以“故障现象 → 排查步骤”为主，技术方向作为辅助筛选。
- 项目同时承担系统学习前后端技术、积累全栈或后端实习作品的目标。

### 16.2 待确认

1. 是否需要对贡献者显示真实姓名、学校和队伍信息？
2. 内容采用何种开源许可协议，转载资料如何标注版权？
3. 首批种子内容由谁提供，预计有多少篇、覆盖哪些方向？
4. 部署预算、预期并发量和是否需要在中国大陆完成备案？

## 17. 当前产品基线

当前已确认或建议作为 MVP 基线：

- 全站内容公开阅读，不建设私有空间。
- 注册用户提交的新增和修改必须经过审核。
- MVP 由项目负责人独立审核，一次通过即可发布。
- 审核期间继续展示最近一次已通过版本。
- 使用用户名或邮箱加密码登录，评论与回复需要登录。
- 只开放经过校验的图片上传，首版不开放工程附件。
- 内容以故障现象和排查步骤为主线，技术方向作为辅助筛选。
- 桌面端只提供富文本编辑，正文保存为结构化 JSON。
- PostgreSQL 作为正式数据库。
- 图片进入对象存储，工程附件首版谨慎开放。
- 不做实时协同，采用版本冲突检测。
- 先以 30–50 篇高频 Debug 条目完成冷启动。
